# Smoothing Summary Generalization Audit

Date: 2026-07-06

## Executive Recommendation

Preferred path: keep `_credit_input_smoothing_summary_row(...)` and
`_curve_input_smoothing_summary_row(...)` as separate helpers for the next
implementation, but feed both from a shared target column/profile mapping
generated by the future shared detail helper.

The two summary helpers are structurally similar enough to share low-level
mechanics, but not cleanly enough to merge immediately without increasing risk.
Their output metric names are target-specific public diagnostic vocabulary, and
their changed-count semantics differ: credit uses valid-pair-gated mismatch
counts while curve uses aligned mismatch counts. That difference may be
intentional, so it should be preserved unless a separate behavior decision and
equality review prove otherwise.

Fallback path: if the shared detail helper is implemented first, centralize only
the column-name assumptions in a target profile and keep the current summary-row
functions reading from that profile. Do not force generic summary metric names in
the first cleanup.

## YAML-Derived Versus Diagnostic-Only Names

Direct or indirect YAML/spec consequences:

- Component score detail columns are derived from YAML component `score.output`
  names, with `raw_` or `smoothed_` diagnostic prefixes.
- Final stance score detail columns are aliases for rule-mapped final score
  outputs exposed by `RuleMappedDiagnosticSpec.final_score_col`.
- Final stance label/strength columns are not consumed by these summary-row
  helpers, but they are produced by detail helpers and should still remain
  spec-derived in the future shared detail helper.

Diagnostic-only aliases:

- `raw_credit_stance_score`, `smoothed_credit_stance_score`,
  `raw_curve_positioning_score`, and `smoothed_curve_positioning_score` are
  display/contract names, not required YAML names.
- `raw_credit_score_change_count`, `smoothed_credit_score_change_count`,
  `raw_curve_score_change_count`, and `smoothed_curve_score_change_count` are
  summary vocabulary choices.
- `score_change_reduction_count`, `score_change_reduction_ratio`,
  `raw_one_day_spike_count`, `smoothed_one_day_spike_count`,
  `one_day_spike_reduction_count`, and `one_day_spike_reduction_ratio` are
  generic diagnostic metric names.

## Feasibility Assessment

Genuinely shared calculations:

- `total_rows`
- valid row count from raw/smoothed final stance score non-null pairs
- component score changed counts
- component score mean absolute differences
- final stance score changed count and ratio
- raw/smoothed final score transition counts
- score-change reduction count and ratio
- raw/smoothed one-day spike counts
- one-day spike reduction count and ratio

Target-specific differences:

- Credit has two component score pairs; curve has three.
- Curve reports `curve_move_driver_score_changed_ratio`; credit has no analogous
  third component metric.
- Credit and curve expose different public metric prefixes.
- Credit uses `_changed_count_for_valid_pairs(...)`; curve uses
  `_changed_count_for_aligned_pairs(...)`.

The changed-count difference is the main blocker to immediate clean merging. For
credit, `_changed_count_for_valid_pairs(...)` computes mismatch and then gates to
rows where both sides are non-null. For curve, `_changed_count_for_aligned_pairs(...)`
counts the full mismatch mask, which treats one-sided missing values as changed.
This may be intentional because curve prepared inputs include filtered move-driver
handling and aligned score comparisons. It may also be accidental historical
drift. The audit cannot safely reclassify it as accidental without runtime
comparison and model-owner approval.

Conclusion:

The helpers can be made YAML/profile-driven now, but a single merged summary-row
helper should wait. The safer implementation is one shared summary engine or
small utility that accepts a target profile and a changed-count mode, while
retaining target-specific wrapper helpers and metric names.

## Recommended Target Profile

A future shared detail helper should emit or expose a target profile with fields
like:

- `final_score_raw_col`
- `final_score_smoothed_col`
- `component_score_pairs`: ordered list of raw/smoothed component score columns
  and metric prefixes
- `changed_count_mode`: `valid_pairs` for credit, `aligned_pairs` for curve
- `final_score_metric_prefix`: `credit_stance_score` or `curve_positioning_score`
- `score_change_metric_prefix`: `credit` or `curve`

With that profile, summary wrappers can stop hard-coding YAML-derived score
column names while preserving public metric names.

## Metric Naming Recommendation

Summary output metric names should remain target-specific for readability and
compatibility. Changing them to generic names would be a public diagnostic output
change and would force downstream users to inspect metadata to understand which
component changed.

Generic names may be useful internally, but the public `summary` and
`window_summary` tables should preserve current names unless a separate API
migration is explicitly requested.

## `compare_smoothing_effect(...)` Dispatch

Current dispatch is already minimal and acceptable:

- target resolution and smoothing-layer checks are shared;
- detail and summary helpers are target-specific;
- `_smoothing_effect_result(...)` is already shared for full-period and window
  summary assembly.

Dispatch cleanup should wait until both detail and summary profile decisions are
implemented together. At that point, it can become:

- one call to a shared detail builder;
- one lookup of a summary profile or wrapper;
- one call to `_smoothing_effect_result(...)`.

Do not fold summary logic into `compare_smoothing_effect(...)`; that would blur
public routing with diagnostic metric calculation.

## Proposed Next Implementation Plan

1. Implement the shared detail helper first, using `RuleMappedDiagnosticSpec` and
   config-derived component score outputs.
2. Introduce a small target profile for public detail/summary column names and
   changed-count mode.
3. Update the existing credit and curve summary-row helpers to consume that
   profile, or wrap a shared summary engine with target-specific helper names.
4. Keep public summary metric names unchanged.
5. Reduce `compare_smoothing_effect(...)` dispatch only after the target profile
   is in place.
6. Validate old-vs-new equality for credit and curve using explicit windows and
   historical-context default windows, with `include_detail=True` and
   `include_detail=False`.
